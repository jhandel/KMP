# 3.9 Multi-Tenancy Architecture and Operations

KMP uses host-based multi-tenancy. Each tenant has its own application database, cache namespace, sessions, queue tables, backups, and document paths. A separate platform datastore holds only tenant registry and provisioning metadata.

## Datastores

| Datasource | Purpose | Migration command |
|------------|---------|-------------------|
| `platform` | Global tenant registry: `tenants`, `tenant_aliases`, `tenant_database_configs`, `tenant_service_configs` | `bin/cake platform:migrate` |
| `tenant` | Current tenant's KMP data: members, roles, workflows, app settings, queue jobs, documents, backups | `bin/cake tenant:migrate --tenant=<slug>` |
| `default` | Legacy/default Cake connection; tenant-aware code should use `tenant` | Legacy only |

Tenant metadata must never be stored in tenant databases. Platform migrations live in `config/PlatformMigrations/`, outside normal tenant migrations, so `tenant:migrate` and `update_database` do not create platform registry tables in tenant databases.

The `platform` datasource must be configured independently with `PLATFORM_DB_*` or `PLATFORM_DATABASE_URL`. It intentionally does not fall back to `DB_DATABASE`, `MYSQL_DB_NAME`, or `DATABASE_URL`, because those identify tenant application databases in many environments.

## Request Routing

1. DNS points each tenant host, such as `tenant-a.kmp.example.org`, to the KMP application.
2. `TenantResolutionMiddleware` runs early, before routing, auth, sessions, CSRF, app settings, or service-principal lookup use tenant data.
3. The resolver looks up the normalized host in the platform registry.
4. Active tenants with the required schema version get a request `TenantContext`.
5. `TenantConnectionFactory` configures the named `tenant` datasource and resets ORM/schema metadata state.

Unknown tenants return a safe 404. Disabled, maintenance, failed, or schema-mismatched tenants return a safe unavailable response before any tenant table is opened.

## Sessions and Authentication

Host-based routing naturally separates browser cookies by host. KMP also uses tenant-specific session names and stores the tenant id/slug in the authenticated session. If an authenticated request arrives with a session from another tenant, the session is destroyed and the user must log in again.

API and service-principal authentication depend on the same resolved tenant context. Do not allow request headers to override host-based tenant resolution unless a future deployment explicitly gates and validates that behavior.

## Cache and Static State Isolation

Tenant-sensitive cache keys are prefixed with the active tenant. This includes schema metadata, app settings, permissions, branch structures, restore status locks, and other cache-backed state. Long-running commands and workers must clear `TenantContext`, table locators, and tenant-prefixed schema metadata between tenants.

## Provisioning Workflow

Run platform migrations before creating tenants:

```bash
cd app
bin/cake platform:migrate
```

Create or update a tenant platform record:

```bash
bin/cake tenant:create example \
  --display-name="Example Kingdom" \
  --primary-host=example.kmp.example.org \
  --database-name=kmp_example \
  --host=mysql.example.internal \
  --username=kmp_example \
  --secret-reference=env:EXAMPLE_DB_PASSWORD \
  --email-config-json='{"transport":{"host":"smtp.example.org","username":"mailer"},"email":{"from":"noreply@example.org"}}' \
  --email-secret-reference=env:EXAMPLE_SMTP_PASSWORD \
  --storage-adapter=s3 \
  --storage-config-json='{"s3":{"bucket":"kmp-example-docs","region":"us-east-1"}}' \
  --storage-secret-reference=env:EXAMPLE_S3_SECRET
```

Preview without writing anything:

```bash
bin/cake tenant:create example --database-name=kmp_example --dry-run --migrate --activate
```

Provisioning sequence for production:

1. Create DNS for the tenant host.
2. Create the tenant database and database user, or run `tenant:create --create-database` where supported.
3. Store database, email, and storage credentials in the platform registry as secret references where possible.
4. Run `tenant:create` to create platform metadata.
5. Run `tenant:migrate --tenant=example`.
6. Seed required app settings and first administrator.
7. Run `tenant:doctor --tenant=example`.
8. Activate the tenant with `tenant:enable example` or `tenant:create --activate` after validation.

## Tenant Operations

| Command | Purpose |
|---------|---------|
| `bin/cake platform:migrate` | Migrate only the platform registry datastore |
| `bin/cake tenant:create <slug>` | Create/update platform tenant metadata and database config |
| `bin/cake tenant:migrate --tenant=<slug>` | Run core and plugin migrations against one tenant database |
| `bin/cake tenant:migrate --all-tenants` | Run tenant migrations for every registered tenant |
| `bin/cake tenant:doctor --tenant=<slug>` | Check status, schema version, DB reachability, and required settings |
| `bin/cake tenant:list` | List platform registry tenants |
| `bin/cake tenant:disable <slug>` | Disable a tenant |
| `bin/cake tenant:maintenance <slug>` | Put a tenant in maintenance mode |
| `bin/cake tenant:enable <slug>` | Reactivate a tenant |

Scheduled tenant commands and queue workers should run with `--tenant=<slug>` or a controlled `--all-tenants` loop. Avoid relying on ambient process state.

## Queues and Workers

Queue tables live in each tenant database. Jobs include tenant context in their payload when enqueued under an active `TenantContext`; workers verify or restore that context before running the task.

Recommended worker model:

```bash
bin/cake queue worker --tenant=example
```

Run separate workers per tenant or supervise an explicit per-tenant loop. A worker must not process jobs for one tenant while connected to another tenant database.

## Backups, Restore, and Storage

Backups and generated files are tenant-scoped. Backup filenames and storage paths include the tenant slug, and restore locks use tenant-prefixed cache keys. Uploaded documents and generated assets should be stored under tenant namespaces such as `tenants/<slug>/...`.

Restore one tenant at a time. Confirm the active tenant context and target database before restoring; never restore a tenant backup into the platform datastore.

## Platform Admin Console

The browser-based onboarding console is available only on hosts listed in `PLATFORM_ADMIN_HOSTS` and only under `/platform-admin`. Tenant hosts cannot serve platform admin routes; platform admin hosts redirect `/` to `/platform-admin`. Local dev may set `PLATFORM_ADMIN_REDIRECT_FROM_HOSTS=localhost,127.0.0.1` so accidental `http://localhost:8080/platform-admin/...` requests redirect to `http://admin.localhost:8080/platform-admin/...` without serving the console on the tenant host.

Run platform migrations and create the first platform admin from the CLI. The seed command creates the admin only if missing, marks it for first-login password rotation, and prints the initial password plus one-time MFA/recovery codes:

```bash
cd app
bin/cake platform:migrate
bin/cake platform_admin:seed \
  --email=platform-admin@example.org \
  --display-name="Root Platform Admin"
```

Store the printed one-time MFA/recovery codes securely; each code is consumed after one successful login or privileged action. After the first login, the browser console requires the admin to choose a new password and rotates the recovery codes again. If access is lost, reset from a trusted shell instead of an unauthenticated browser endpoint:

```bash
bin/cake platform_admin:reset_password root@example.org
```

Platform admin accounts live in the `platform` datasource and are separate from tenant `members`.

Platform email is also separate from tenant email. Store password-reset, recovery, and security-notification transport metadata in `platform_service_configs` with `service_name=email`, `config_key=default`, and secret references such as `env:PLATFORM_SMTP_PASSWORD` or platform-managed `managed:platform/email/default`. Do not reuse tenant `tenant_service_configs` for platform admin mail.

For containerized cloud deployments where admins cannot SSH into a host to add per-tenant environment variables, configure `PLATFORM_SECRET_KEY` once at deployment time and then store tenant database/email/storage secrets from the platform console. Managed secrets are encrypted in the platform datastore and referenced as `managed:tenant/<id>/...`; they are never displayed after saving. Keep `PLATFORM_SECRET_KEY` stable and back it up securely. Changing it without re-encrypting stored secrets will make managed secrets unreadable.

The console currently provides:

- tenant list/detail and doctor checks,
- tenant creation with database, email, and storage metadata,
- status changes with action verification,
- encrypted backup creation,
- restore into a new database followed by cutover,
- platform audit review with hash-chained event records.

The console deliberately does not provide free-form SQL/phpMyAdmin functionality. If database diagnostics are needed later, add a constrained library of parameterized read-only checks instead of arbitrary SQL execution.

## Tenant Runtime Configuration

Tenant-specific values that were historically process-wide environment variables should be represented in platform metadata instead of shared `.env` values. `tenant_database_configs` stores tenant database targets, while `tenant_service_configs` stores per-tenant runtime metadata for services such as email and document storage.

Supported first-release service configs:

| Service | Metadata shape | Secret reference behavior |
|---------|----------------|---------------------------|
| `email` | `{"transport": {...}, "email": {...}}` merged into `EmailTransport.default` and `Email.default` | `env:NAME` or `managed:NAME` becomes the email transport password |
| `storage` | Document storage metadata such as `{"azure": {...}}` or `{"s3": {...}}` | `env:NAME` or `managed:NAME` becomes the Azure connection string or S3 secret based on the adapter |

Examples:

```bash
bin/cake tenant:create ansteorra \
  --database-name=kmp_ansteorra \
  --secret-reference=env:ANSTEORRA_DB_PASSWORD \
  --email-config-json='{"transport":{"host":"smtp.ansteorra.example","username":"mailer"},"email":{"from":"noreply@ansteorra.example"}}' \
  --email-secret-reference=env:ANSTEORRA_SMTP_PASSWORD \
  --storage-adapter=azure \
  --storage-config-json='{"azure":{"container":"kmp-documents"}}' \
  --storage-secret-reference=env:ANSTEORRA_AZURE_STORAGE_CONNECTION_STRING
```

Do not create global `.env` keys for every tenant service value unless the key is referenced from platform metadata. Shared keys like `AZURE_STORAGE_CONNECTION_STRING` and `EMAIL_SMTP_PASSWORD` are deployment defaults only; tenant production values should use tenant-specific references such as `env:ANSTEORRA_AZURE_STORAGE_CONNECTION_STRING` or managed platform references created through the console.

## Verification Checklist

- `config/PlatformMigrations/` contains platform registry migrations; `config/Migrations/` does not.
- `platform` datasource uses only `PLATFORM_*` env vars or `PLATFORM_DATABASE_URL`.
- Tenant database, email, and storage service settings are stored in platform metadata with tenant-specific secret references.
- `bin/cake platform:migrate` succeeds before tenant provisioning.
- `bin/cake tenant:create --dry-run` previews changes without creating records.
- `bin/cake tenant:migrate --tenant=<slug>` migrates tenant tables only.
- `bin/cake tenant:doctor --tenant=<slug>` passes before activation.
- Tenant A cannot read Tenant B data, cache keys, queue jobs, backup files, or document paths.
