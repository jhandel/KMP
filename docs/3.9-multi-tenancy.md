# 3.9 Multi-Tenancy Architecture and Operations

KMP uses host-based multi-tenancy. Each tenant has its own application database, cache namespace, sessions, queue tables, backups, and document paths. A separate platform datastore holds only tenant registry and provisioning metadata.

For the control-plane orchestration model choice (platform-native state machine vs generic workflow engine reuse), see [Platform Operation Engine Decision ADR](architecture/platform-operation-engine-decision.md).
For canonical platform operation lifecycle states/transitions, idempotency, lock semantics, and all-tenant child-job orchestration, see [Platform Operation State Machine](architecture/platform-operation-state-machine.md).
For production-safe admin/console command execution in Azure multi-replica environments (break-glass policy, gateway model, approvals, and targeting modes), see [Deployment: Production Command Strategy](deployment/production-command-strategy.md).
For cross-pod cache/config/schema invalidation after tenant config, cutover, secret, migration, or status changes, see [Cross-Pod Invalidation Design](architecture/cross-pod-invalidation-design.md).
For deployment-time platform-first sequencing, active-tenant discovery, batch ordering, retry/resume semantics, and schema-compatibility serving gates, see [Deployment Migration Strategy](deployment/deployment-migration-strategy.md).
For production go/no-go release gating across multi-tenancy hardening workstreams, use the [Deployment Production Readiness Checklist](deployment/production-readiness-checklist.md).

## Datastores

| Datasource | Purpose | Migration command |
|------------|---------|-------------------|
| `platform` | Global tenant registry: `tenants`, `tenant_aliases`, `tenant_database_configs`, `tenant_service_configs` | `bin/cake platform:migrate` |
| `tenant` | Current tenant's KMP data: members, roles, workflows, app settings, queue jobs, documents, backups | `bin/cake tenant:migrate --tenant=<slug>` |
| `default` | Legacy/default Cake connection; tenant-aware code should use `tenant` | Legacy only |

Tenant metadata must never be stored in tenant databases. Platform migrations live in `config/PlatformMigrations/`, outside normal tenant migrations, so `tenant:migrate` and `update_database` do not create platform registry tables in tenant databases.

The `platform` datasource must be configured independently with `PLATFORM_DB_*` or `PLATFORM_DATABASE_URL`. It intentionally does not fall back to `DB_DATABASE`, `MYSQL_DB_NAME`, or `DATABASE_URL`, because those identify tenant application databases in many environments.

## Request Routing

1. DNS points each tenant host, such as `tenant-a.kmp.example.org`, to the KMP application.
2. `TenantResolutionMiddleware` runs early, before routing, auth, sessions, CSRF, app settings, or service-principal lookup use tenant data.
3. The resolver looks up the normalized host in the platform registry.
4. Active tenants with the required schema version get a request `TenantContext`.
5. `TenantConnectionFactory` configures the named `tenant` datasource and resets ORM/schema metadata state.

Unknown tenants return a safe 404. Disabled, maintenance, failed, or schema-mismatched tenants return a safe unavailable response before any tenant table is opened.

## Sessions and Authentication

Host-based routing naturally separates browser cookies by host. KMP also uses tenant-specific session names and stores the tenant id/slug in the authenticated session. If an authenticated request arrives with a session from another tenant, the session is destroyed and the user must log in again.

API and service-principal authentication depend on the same resolved tenant context. Do not allow request headers to override host-based tenant resolution unless a future deployment explicitly gates and validates that behavior.

## Cache and Static State Isolation

Tenant-sensitive cache keys are prefixed with the active tenant. This includes schema metadata, app settings, permissions, branch structures, and other cache-backed state. Long-running commands and workers must clear `TenantContext`, table locators, and tenant-prefixed schema metadata between tenants.

## Provisioning Workflow

Run platform migrations before creating tenants:

```bash
cd app
bin/cake platform:migrate
```

Create or update a tenant platform record:

```bash
bin/cake tenant:create example \
  --display-name="Example Kingdom" \
  --primary-host=example.kmp.example.org \
  --database-name=kmp_example \
  --host=mysql.example.internal \
  --username=kmp_example \
  --secret-reference=env:EXAMPLE_DB_PASSWORD \
  --email-config-json='{"transport":{"host":"smtp.example.org","username":"mailer"},"email":{"from":"noreply@example.org"}}' \
  --email-secret-reference=env:EXAMPLE_SMTP_PASSWORD \
  --storage-adapter=s3 \
  --storage-config-json='{"s3":{"bucket":"kmp-example-docs","region":"us-east-1"}}' \
  --storage-secret-reference=env:EXAMPLE_S3_SECRET
```

Preview without writing anything:

```bash
bin/cake tenant:create example --database-name=kmp_example --dry-run --migrate --activate
```

Provisioning sequence for production:

1. Create DNS for the tenant host.
2. Create the tenant database and database user, or run `tenant:create --create-database` where supported.
3. Store database, email, and storage credentials in the platform registry as secret references where possible.
4. Run `tenant:create` to create platform metadata.
5. Run `tenant:migrate --tenant=example`.
6. Seed required app settings and first administrator.
7. Run `tenant:doctor --tenant=example`.
8. Activate the tenant with `tenant:enable example` or `tenant:create --activate` after validation.

## Tenant Operations

| Command | Purpose |
|---------|---------|
| `bin/cake platform:migrate` | Migrate only the platform registry datastore |
| `bin/cake deployment:migrate --wait --drive-worker` | Platform-first deployment migration orchestrator: runs `platform:migrate`, snapshots active tenants, enqueues/monitors `tenant_migrate` child jobs |
| `bin/cake tenant:create <slug>` | Create/update platform tenant metadata and database config |
| `bin/cake tenant:migrate --tenant=<slug>` | Run core and plugin migrations against one tenant database |
| `bin/cake tenant:migrate --all-tenants` | Run tenant migrations for every registered tenant |
| `bin/cake tenant:doctor --tenant=<slug>` | Check status, schema version, DB reachability, and required settings |
| `bin/cake tenant:list` | List platform registry tenants |
| `bin/cake tenant:disable <slug>` | Disable a tenant |
| `bin/cake tenant:maintenance <slug>` | Put a tenant in maintenance mode |
| `bin/cake tenant:enable <slug>` | Reactivate a tenant |
| `bin/cake tenant_operation:worker --once --limit=5` | Execute durable platform `tenant_operation_jobs` outside request lifecycle |

Scheduled tenant commands and queue workers should run with `--tenant=<slug>` or a controlled `--all-tenants` loop. Avoid relying on ambient process state.

For control-plane operations recorded in `tenant_operation_jobs`, run the dedicated worker command (`tenant_operation:worker`) under a supervised runner. The worker acquires lease ownership, updates progress/error/result payloads, and retries retryable failures using `input.max_attempts` (default 3) before moving jobs to `failed`.

Deployment migration orchestration uses a parent `tenant_migrate_all` operation and per-tenant `tenant_migrate` child jobs sharing one correlation id. Child rows are explicitly linked through `tenant_operation_jobs.parent_tenant_operation_job_id` (with legacy input fallback for older records). Child result payloads include schema-before/schema-after, migration scopes, and duration metadata; child error payloads include retryability and schema context. In `--wait` mode, `deployment:migrate` can either monitor an external worker or process child jobs in-process with `--drive-worker`. Failed children can be resumed by rerunning with `--resume-parent-id=<id>`.

### Structured tenant logging contract

Tenant-aware platform operations and middleware error paths must emit the following structured context fields whenever available:

- `request_id`
- `tenant_id`
- `tenant_slug`
- `operation_id`
- `platform_admin_id`
- `correlation_id`
- `operation_image`
- `operation_version`

### Platform audit ↔ operation job correlation contract

Platform audit events and `tenant_operation_jobs` are linked through metadata fields:

- `metadata.operation_id` stores the `tenant_operation_jobs.id` value when an audit event is emitted from an operation path.
- `metadata.correlation_id` must match `tenant_operation_jobs.operation_correlation_id` for the same operation lifecycle step.
- Correlation IDs must be immutable per operation and reused across retry/cancel/audit records created during that request path.

The platform admin UI exposes this linkage in both surfaces:

- **Operation Queue** supports filtering by correlation id and links directly to filtered audit events.
- **Platform Audit** supports filtering by correlation/operation id and shows linked operation jobs.

Sensitive values must be redacted before persistence or emission to app logs. This includes passwords, tokens, secret values, credentials, and connection-string style keys (including nested metadata fields).

### Tenant metrics and health signals

KMP emits process-local tenant control-plane metrics for resolution and operation visibility. Metric labels are intentionally bounded and must never include tenant hosts, slugs, emails, request payloads, or secret-derived values.

| Metric | Type | Labels | Description |
|---|---|---|---|
| `kmp_tenant_resolution_duration_ms` | Histogram | `outcome` (`success`, `unknown_tenant`, `inactive_tenant`, `draining_tenant`, `schema_mismatch`, `empty_host`, `error`) | End-to-end tenant resolution latency in middleware/resolver flow. |
| `kmp_tenant_registry_query_total` | Counter | `outcome` (`cache_hit`, `cache_stale`, `alias_hit`, `primary_hit`, `miss`) | Tenant registry lookup outcome volume. |
| `kmp_tenant_operation_queue_depth` | Gauge | `queue=runnable` | Current runnable operation queue depth sampled by worker loop. |
| `kmp_tenant_operation_duration_ms` | Histogram | `operation`, `outcome` (`success`, `failed`, `cancelled`) | Per-attempt operation execution duration. |
| `kmp_tenant_operation_outcome_total` | Counter | `operation`, `outcome` | Per-attempt operation outcome counts. |
| `kmp_tenant_health_signal_total` | Counter | `signal` | Resolution and operation health events (`resolution_*`, `operation_*`). |
| `kmp_tenant_connection_pool_connections` | Gauge | `driver`, `state` (`active`, `idle`, `waiting`, `total`, `max`) | Latest tenant DB pool sample counts (PostgreSQL probes; unsupported drivers are tracked as probe errors). |
| `kmp_tenant_connection_pool_saturation_ratio` | Gauge | `driver` | Active/max ratio from the latest tenant pool sample (0.0-1.0). |
| `kmp_tenant_connection_pool_saturation_total` | Counter | `driver`, `risk` (`normal`, `high`, `critical`) | Count of sampled saturation classifications for trend and alert dashboards. |
| `kmp_tenant_connection_pool_timeout_total` | Counter | `driver`, `phase` (`connect`, `probe`, `query`, `other`) | Timeout signals when observable during pool sampling/connection probing. |
| `kmp_tenant_connection_pool_probe_error_total` | Counter | `driver`, `error` (`unsupported_driver`, `permission_denied`, `timeout`, `query_failed`, `other`) | Probe failure outcomes for visibility when pool sampling is unavailable or failing. |

Health surface integration:

- `/health` now includes `tenant_metrics` (current in-process snapshot) and `tenant_health` (platform summary counters) alongside existing DB/cache checks.
- `/platform-admin` now includes a **Tenant Health** panel with tenant-level migration/backlog/failure/drain/invalidation signals for platform operators.
- `/health` now includes `tenant_connection_pool` with dashboard-friendly aggregates for pool usage, saturation, probe errors, and timeout indicators.
- High/critical saturation and probe failures are emitted to the existing `app.performance` logging scope for centralized operational alerting.
- These counters are for operational diagnostics and trend monitoring; use platform audit for tenant-specific traces.

Expected operating ranges (adjust by environment scale and active rollout windows):

| Signal | Healthy range | Investigate when |
|---|---|---|
| Migration outdated tenants | `0` | `>0` outside active deployment windows |
| Operation backlog (`queued`/`approved`/`running`/`hold`/`blocked`) | `0-5` steady | sustained growth or sudden spikes |
| Recent failures (24h) | `0` | repeated or correlated failures |
| Draining tenants | `0` | draining persists past planned maintenance |
| Invalidation lag p95 | `<300s` | `>=300s` or rising trend |

## Queues and Workers

Queue tables live in each tenant database. Jobs include tenant context in their payload when enqueued under an active `TenantContext`; workers verify or restore that context before running the task.

Recommended worker model:

```bash
bin/cake queue worker --tenant=example
```

Run separate workers per tenant or supervise an explicit per-tenant loop. A worker must not process jobs for one tenant while connected to another tenant database.

## Backups, Restore, and Storage

Backups and generated files are tenant-scoped. Backup filenames and storage paths include the tenant slug, and destructive platform operations are serialized by platform lease table `tenant_operation_locks` (one active lock per tenant with stale-lock recovery). Uploaded documents and generated assets should be stored under tenant namespaces such as `tenants/<slug>/...`.

Restore one tenant at a time. Confirm the active tenant context and target database before restoring; never restore a tenant backup into the platform datastore.

## Platform Admin Console

The browser-based onboarding console is available only on hosts listed in `PLATFORM_ADMIN_HOSTS` and only under `/platform-admin`. Tenant hosts cannot serve platform admin routes; platform admin hosts redirect `/` to `/platform-admin`. Local dev may set `PLATFORM_ADMIN_REDIRECT_FROM_HOSTS=localhost,127.0.0.1` so accidental `http://localhost:8080/platform-admin/...` requests redirect to `http://admin.localhost:8080/platform-admin/...` without serving the console on the tenant host.

Run platform migrations and create the first platform admin from the CLI. The seed command creates the admin only if missing and prints the initial password. Login and privileged-action verification codes are emailed to the platform admin address:

```bash
cd app
bin/cake platform:migrate
bin/cake platform_admin:seed \
  --email=platform-admin@example.org \
  --display-name="Root Platform Admin"
```

Store the printed initial password securely and rotate it on first use unless the account was intentionally seeded with `--no-require-change` for local development. Verification codes are short-lived email codes, not CLI-printed recovery codes. In development, these emails are captured in Mailpit. If access is lost, reset from a trusted shell instead of an unauthenticated browser endpoint:

```bash
bin/cake platform_admin:reset_password root@example.org
```

Platform admin accounts live in the `platform` datasource and are separate from tenant `members`. The local reset scripts seed `platform-admin@localhost.test` with password `TestPassword` and the `break_glass` role for `/platform-admin`; tenant users such as `admin@amp.ansteorra.org` are separate member accounts. See the [Platform Admin UI and Operations Guide](platform-admin-ui.md) for the full route, role, action-code, command-catalog, backup/restore, and audit behavior.

Platform email is also separate from tenant email. Store sign-in verification, privileged-action verification, and security-notification transport metadata in `platform_service_configs` with `service_name=email`, `config_key=default`, and secret references such as `env:PLATFORM_SMTP_PASSWORD` or platform-managed `managed:platform/email/default`. Do not reuse tenant `tenant_service_configs` for platform admin mail.

For containerized cloud deployments where admins cannot SSH into a host to add per-tenant environment variables, configure `PLATFORM_SECRET_KEY` once at deployment time and then store tenant database/email/storage secrets from the platform console. Managed secrets are encrypted in the platform datastore and referenced as `managed:tenant/<id>/...`; they are never displayed after saving. Keep `PLATFORM_SECRET_KEY` stable and back it up securely. Changing it without re-encrypting stored secrets will make managed secrets unreadable.

Database credential updates submitted from `/platform-admin/tenants/{slug}/secrets` are executed as a queued `tenant_rotate_db_secret` operation. The worker performs staged execution:

1. prepare rotation context,
2. update `tenant_database_configs.secret_reference` to the new managed reference,
3. invalidate local connection/secret/runtime caches,
4. verify tenant DB connectivity,
5. rollback to previous reference if verification fails.

Successful rotations publish `tenant_secret_rotated` invalidation events; failed verification publishes `tenant_secret_rotation_rollback`.

The console currently provides:

- tenant list/detail and doctor checks,
- deployment migration dashboard for `tenant_migrate_all` parent runs with:
  - platform migration stage and child-state counts,
  - per-tenant schema/result/error summaries,
  - hold/failure indicators and `Resume` affordance for held/blocked parents,
  - child linkage sourced from `tenant_operation_jobs.parent_tenant_operation_job_id` (with legacy fallback),
- JSON API projection at `/platform-admin.json` (or `Accept: application/json`) with deployment migration filters:
  - `migration_state`,
  - `migration_tenant_state`,
  - `migration_correlation`,
  - `migration_limit`,
- tenant creation with database, email, and storage metadata,
- status changes with action verification,
- encrypted backup creation,
- restore into a new database followed by cutover,
- platform audit review with hash-chained event records.

The console deliberately does not provide free-form SQL/phpMyAdmin functionality. If database diagnostics are needed later, add a constrained library of parameterized read-only checks instead of arbitrary SQL execution.

## Tenant Runtime Configuration

Tenant-specific values that were historically process-wide environment variables should be represented in platform metadata instead of shared `.env` values. `tenant_database_configs` stores tenant database targets, while `tenant_service_configs` stores per-tenant runtime metadata for services such as email and document storage.

Supported first-release service configs:

| Service | Metadata shape | Secret reference behavior |
|---------|----------------|---------------------------|
| `email` | `{"transport": {...}, "email": {...}}` merged into `EmailTransport.default` and `Email.default` | `env:NAME` or `managed:NAME` becomes the email transport password |
| `storage` | Document storage metadata such as `{"azure": {...}}` or `{"s3": {...}}` | `env:NAME` or `managed:NAME` becomes the Azure connection string or S3 secret based on the adapter |

Examples:

```bash
bin/cake tenant:create ansteorra \
  --database-name=kmp_ansteorra \
  --secret-reference=env:ANSTEORRA_DB_PASSWORD \
  --email-config-json='{"transport":{"host":"smtp.ansteorra.example","username":"mailer"},"email":{"from":"noreply@ansteorra.example"}}' \
  --email-secret-reference=env:ANSTEORRA_SMTP_PASSWORD \
  --storage-adapter=azure \
  --storage-config-json='{"azure":{"container":"kmp-documents"}}' \
  --storage-secret-reference=env:ANSTEORRA_AZURE_STORAGE_CONNECTION_STRING
```

Do not create global `.env` keys for every tenant service value unless the key is referenced from platform metadata. Shared keys like `AZURE_STORAGE_CONNECTION_STRING` and `EMAIL_SMTP_PASSWORD` are deployment defaults only; tenant production values should use tenant-specific references such as `env:ANSTEORRA_AZURE_STORAGE_CONNECTION_STRING` or managed platform references created through the console.

## Verification Checklist

- `config/PlatformMigrations/` contains platform registry migrations; `config/Migrations/` does not.
- `platform` datasource uses only `PLATFORM_*` env vars or `PLATFORM_DATABASE_URL`.
- Tenant database, email, and storage service settings are stored in platform metadata with tenant-specific secret references.
- `bin/cake platform:migrate` succeeds before tenant provisioning.
- `bin/cake tenant:create --dry-run` previews changes without creating records.
- `bin/cake tenant:migrate --tenant=<slug>` migrates tenant tables only.
- `bin/cake tenant:doctor --tenant=<slug>` passes before activation.
- Tenant A cannot read Tenant B data, cache keys, queue jobs, backup files, or document paths.
- Production release sign-off is completed via [Deployment Production Readiness Checklist](deployment/production-readiness-checklist.md).
